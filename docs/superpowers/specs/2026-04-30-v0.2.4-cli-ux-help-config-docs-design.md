# v0.2.4 CLI UX Help And Config Docs Design

## Summary

v0.2.4 focuses on user-facing command-line experience quality: clearer help output, completion/help parity, better config file documentation, and tighter behavior for common failure paths. This release also includes targeted UX bugfixes (duplicate subscription tags), aggressive command-structure cleanup, and an allowed runtime/daemon rewrite so long as CLI contracts remain compatible.

## Release Goal

Make `subcli` easier to discover, easier to configure, and harder to misuse by ensuring users see accurate command guidance at the CLI, can rely on completion hints, and have a single clear reference for `config.yaml` responsibilities.

## In Scope

1. Help/usage consistency improvements for root and selected subcommands.
2. Bash completion parity with implemented options and subcommands.
3. New config file reference document and README link updates.
4. Duplicate tag normalization for `sub add` and `sub edit` flows.
5. Export failure-contract tests for empty selection and zero parsed nodes.
6. Command-surface decomposition that reduces duplication and keeps behavior testable.
7. Runtime/daemon subsystem rewrite where needed to improve reliability and maintainability while preserving public command contracts.

## Out Of Scope

1. No release CI automation changes.
2. No config format breakage or migration requirement.
3. No intentional CLI command removal or incompatible flag rename.

## Current Baseline And Gaps

### Baseline

- Help text exists for root and subcommands in `src/main.cpp` and is covered by smoke tests.
- Completion script exists in `src/cli_completion.cpp` and is covered by unit checks.
- Documentation includes `README.md`, `README.subcli.md`, and `docs/profile-schema.md`.

### Gaps

- Completion misses implemented options (`export --strict-capabilities --explain-policy --json`) and `profile explain` subcommand.
- Completion config key suggestions are narrower than actual supported key space.
- No dedicated `config.yaml` reference document that clearly separates app settings from profile policy settings.
- Duplicate tags can be persisted from `sub add`/`sub edit` input.
- Export failure contracts for empty selection/zero-node paths are under-tested at direct CLI behavior level.

## Design

### 1) Help Experience

Keep existing manual help style in `src/main.cpp`, but tighten consistency rules:

- Root help command summaries must match real command surface.
- Subcommand usage lines (`print*SubcommandUsage`) must align with parser-supported flags.
- `config` help must remain aligned with actual supported key prefixes.
- `profile`/`export` help must remain aligned with real parser flags and examples.

This is polish, not behavior redesign. Existing phrasing style is preserved where possible.

### 2) Completion Parity

Update `src/cli_completion.cpp` so suggestions reflect what parser/help already supports:

- `profile` adds `explain`.
- `export` adds `--strict-capabilities`, `--explain-policy`, `--json`.
- `config` adds missing key suggestions used by real handlers/help:
  - `profile_path`
  - `grouping.region_rules.`
  - `assets.paths.`
  - `assets.urls.`
  - `templates.` key pattern hints

Completion remains Bash-only for this release.

### 3) Config File Documentation

Add `docs/config-file.md` as the main `config.yaml` reference:

- What `config.yaml` controls vs what `profile.json` controls.
- Path resolution rules (config-dir-relative persisted paths vs CLI arg cwd-relative behavior).
- Workspace precedence and effective path behavior (`--workspace` > `SUBCLI_WORKSPACE` > discovery/persisted/default flow).
- Common keys and prefixes with concise examples.
- Minimal full example `config.yaml`.
- Typical command workflow (`config list/get/set/remove`, plus handoff to `template` and `profile`).

Update `README.md` and `README.subcli.md` documentation sections to link this new reference.

### 4) Duplicate Tag Normalization

Normalize tags from both add/edit paths before persistence:

- Preserve first-seen order.
- Drop exact duplicates.
- Keep behavior deterministic.

Normalization must apply to:

- Combined `sub add --tag` + `--tags` inputs.
- `sub edit --tags` replacement input.

No format migration is required; existing stored duplicates are tolerated until next edit/write flow.

### 5) Export Failure Contracts

Add tests for direct CLI contracts:

- Export fails with clear message when selection filters produce no subscriptions.
- Export fails with clear message when selected subscriptions parse into zero usable nodes.

Coverage should verify user-visible error text and exit codes through existing smoke and/or targeted tests.
Expected baseline contract strings in current implementation:

- `no subscriptions selected for export (check --sub/--tag filters)` with exit code `1`.
- `no nodes parsed from enabled subscriptions` with exit code `1`.

### 6) Command And Runtime/Daemon Decomposition

Perform decomposition that directly serves this release scope and runtime/daemon reliability:

- Introduce shared constants/helpers for command help/completion parity.
- Introduce focused tag normalization helper used by `sub add` and `sub edit`.
- Extract daemon mode parsing/execution into focused units to reduce `doDaemonCommand` branching complexity.
- Extract runtime process-state and daemon process-state primitives into clearer internal units if needed.

Broad refactor is allowed for touched areas (`daemon`, runtime lifecycle helpers, selected CLI command surfaces), but all behavior changes must be locked by tests.

## Data Flow And Behavior Notes

### Help/Completion Consistency Flow

1. Parser flags in command handlers define capability.
2. Help text documents capability.
3. Completion suggests capability.
4. Tests assert parity signals to catch drift.

v0.2.4 strengthens steps 2-4 without changing step 1 semantics.

### Tag Normalization Flow

1. Gather tags from CLI flags.
2. Normalize into first-seen unique list.
3. Persist normalized list to subscription store.
4. Show normalized tags through `sub list` and `sub list --json` output.

## Error Handling Strategy

- Continue existing CLI error contract style (`stderr` message + non-zero code).
- Runtime/daemon internals may be rewritten, but error semantics at CLI boundary stay stable.
- New/updated tests pin expected behavior for changed paths.

## Testing Strategy

### Unit/Component

- Completion text assertions for newly added command/options.
- Tag normalization behavior tests for add/edit parsing/persistence paths.
- Runtime lifecycle tests for rewritten process control paths.
- Daemon lifecycle and cycle-summary tests for rewritten daemon execution paths.

### Smoke/CLI

- Help and completion parity checks remain and are extended for new expectations.
- Export empty-selection and zero-node-failure contract checks are added/strengthened.

### Build Verification

Use project standard checks:

```bash
cmake -S . -B build && cmake --build build -j
ctest --test-dir build --output-on-failure
```

## Acceptance Criteria

1. `subcli completion bash` includes `profile explain` and export parity options.
2. Help output and parser-supported options remain consistent for touched commands.
3. New `docs/config-file.md` exists and is linked from both `README.md` and `README.subcli.md`.
4. Duplicate tags are normalized during add/edit before persistence.
5. Export failure contracts for empty selection and zero usable nodes are covered by tests.
6. Build and test pass with standard commands.

## Risks And Mitigations

- Risk: Help/completion drift reappears.
  - Mitigation: add targeted assertions in existing test suites.
- Risk: Small refactor grows scope.
  - Mitigation: stage refactor in test-first slices with command-level compatibility checks.
- Risk: Behavior surprises from tag dedupe.
  - Mitigation: preserve order, dedupe exact matches only, document behavior in release notes.
- Risk: Runtime/daemon rewrite introduces process-management regressions.
  - Mitigation: preserve and expand lifecycle tests (`start/inspect/stop`, stale state handling, daemon cycle summaries, restart decisions).

## Implementation Boundaries

- Prioritize user-visible UX correctness over internal restructuring.
- Keep backward compatibility for current command syntax and config file semantics.
- Runtime/daemon internals may be aggressively rewritten if tests prove contract stability.
- Treat this release as a quality pass for discoverability, operational clarity, and process-management reliability.
