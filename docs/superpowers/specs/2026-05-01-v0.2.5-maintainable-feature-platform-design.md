# v0.2.5 Maintainable Feature Platform Design

## Summary

v0.2.5 is a large quality-and-capability release. The release goal is to make all core user workflows usable end-to-end while changing the code shape so commands, config keys, export targets, templates, and capability rules are not hard-coded in scattered branches. The design favors centralized registries, smaller command modules, service-layer logic, and direct tests that prove behavior remains stable while features become easier to add, remove, or change.

## Release Goals

1. Make subscription lifecycle operations complete enough for daily use: add, import, export, validate/check, update, prune, and batch edit.
2. Reduce hard-coded command/config/target/template/capability lists by introducing explicit registries as source-of-truth data structures.
3. Improve diagnostics so users and automation can understand workspace, config, template, asset, core, cache, subscription, and export readiness state.
4. Keep exporter behavior compatible while making new target/protocol work follow a clear extension path.
5. Preserve build/test reliability and add release validation workflow coverage for build, tests, and package generation.

## Non-Goals

1. No external plugin ABI or dynamic shared-library plugin system in v0.2.5.
2. No incompatible config format migration unless a compatibility-safe default path exists.
3. No GUI or daemon-first product pivot; native config export remains the primary workflow.
4. No broad rewrite of parsers/exporters unrelated to registry boundaries or selected feature work.

## Current Baseline

- `master` includes v0.2.4 CLI UX work, config docs, tag normalization, export failure tests, and runtime/daemon helper decomposition.
- `src/main.cpp` still owns much command orchestration and contains repeated config-key, help, export, and diagnostic branching.
- `CMakeLists.txt` still has package metadata versioned as `0.1.0`, while project docs discuss later release work.
- Existing install/package metadata is locally usable, but there is no checked-in release-validation workflow.
- `docs/config-file.md` documents current config keys and path rules, but those docs are still manually maintained rather than generated or registry-checked.

## Architecture

### 1) Registry Layer

Add small, static C++ registries. These are compile-time source-of-truth tables, not runtime plugins.

Planned registries:

- `CommandRegistry`: command names, aliases, summaries, usage examples, nested subcommands, and completion groups.
- `ConfigKeyRegistry`: config key names, type, default behavior, getter/setter/remove callbacks, path-resolution behavior, and help text.
- `TargetRegistry`: export target IDs, display names, output filenames, template slots, core path key, and checker type.
- `TemplateRegistry`: built-in template paths, target associations, tun/non-tun variants, validation expectations, and config key mapping.
- `CapabilityRegistry`: protocol/target support levels, warning codes, degradation notes, and strict-capability blocking behavior.

Each registry should have a narrow API, for example `allConfigKeys()`, `findTarget(id)`, or `capabilityFor(target, node)`. Command handlers should ask registries for data rather than keeping duplicate string lists.

### 2) Command Modules

Split command execution into focused modules under `src/commands/` and `include/subcli/commands/`.

Initial modules:

- `sub_command`: subscription CLI parsing and display.
- `config_command`: registry-backed config list/get/set/remove.
- `export_command`: target selection, profile selection, export orchestration, JSON output.
- `doctor_command`: readiness checks and JSON/text diagnostic output.
- `completion_command`: completion generation from command/config/target registries.

`src/main.cpp` remains the program entrypoint and global option router. It should not contain long feature-specific branches after extraction.

### 3) Service Layer

Put reusable behavior behind service APIs that command modules call.

Initial services:

- `SubscriptionService`: import/export/list/filter/batch-edit/prune/check operations.
- `ConfigService`: typed registry-backed config mutation and validation.
- `ExportService`: selected subscription fetching/parsing/preprocessing/export target execution.
- `DiagnosticService`: doctor probes for workspace, config, templates, assets, profiles, cores, cache, subscriptions, and export readiness.

Services should be testable without invoking the CLI binary. CLI smoke tests remain for user-facing contracts.

## Feature Scope

### Subscription Lifecycle

Add or complete these user-facing commands:

- `sub import --file PATH [--format auto|subcli-yaml|subcli-json|uri-list] [--merge|--replace]`
- `sub export --file PATH [--format subcli-yaml|subcli-json] [--tag TAG] [--group GROUP] [--enabled true|false]`
- `sub check [ID_OR_NAME...] [--tag TAG] [--json]`
- `sub prune [--failed-days N] [--disabled] [--dry-run]`
- `sub edit --tag TAG --set-group GROUP` style batch-safe paths where existing syntax allows clean extension.

Import/export formats should preserve supported subscription fields: id, name, url, enabled, group, tags, format hint, headers, timeouts, retry, update interval, priority, and cache metadata where appropriate. Import should reject ambiguous duplicate IDs unless `--merge` gives deterministic conflict behavior.

### Diagnostics

Extend `doctor` around structured checks:

- workspace source and resolved paths.
- config key validity and unknown-key detection.
- profile availability and validation.
- template presence and target/template mapping.
- asset file/url status.
- core discovery and configured core paths.
- subscription store readability, cache existence, stale/error state.
- export readiness per target: profile, templates, assets, selected subscription count, parseable node count, capability summary.

`doctor --json` should use stable code strings. Text mode can remain concise and human-oriented.

### Target And Protocol Extension Path

Do not add many protocols by hand in unrelated branches. v0.2.5 should define the extension path and prove it with targeted improvements.

Required extension checklist for any protocol or target change:

1. parser fixture and parser test.
2. protocol registry/capability registry entry.
3. exporter support or explicit degradation/unsupported warning.
4. strict-capability behavior test.
5. profile explain/export JSON evidence test.
6. docs update or generated registry check.

Add exactly one small protocol or protocol-field improvement as a validation slice. The implementation plan must choose the slice from current parser/exporter gaps and apply the full extension checklist above.

### Release Validation

Add `.github/workflows/release-validation.yml` with a minimal workflow:

- configure with `cmake -S . -B build`.
- build with `cmake --build build -j`.
- test with `ctest --test-dir build --output-on-failure`.
- package with `cmake --build build --target package`.
- upload package artifact on workflow dispatch and tag pushes.

Update CMake project/package version so generated packages match the release stream.

## Data Flow

### Config Key Flow

1. `ConfigKeyRegistry` defines key metadata, type, setter, getter, remove behavior, and docs text.
2. `config list/get/set/remove` uses the registry instead of handwritten key branches.
3. completion and help use the registry key list.
4. docs tests compare `docs/config-file.md` coverage against registry keys or generate a stable key reference section.

### Export Target Flow

1. `TargetRegistry` defines target metadata.
2. CLI target parsing resolves IDs through the registry.
3. template resolution asks `TemplateRegistry` for target/template slots.
4. export orchestration loops selected target descriptors instead of branching per target where possible.
5. capability summaries come from `CapabilityRegistry`.

### Subscription Import Flow

1. read input file.
2. detect or honor format.
3. parse into `Subscription` records.
4. validate required fields and normalize tags/headers.
5. merge or replace using deterministic conflict rules.
6. persist through existing store path.
7. report created/updated/skipped/rejected counts.

### Doctor Flow

1. build an environment/workspace snapshot.
2. run independent diagnostic probes.
3. collect results with stable code, severity, subject, message, and optional path.
4. emit text or JSON.
5. return zero for informational/warning-only findings and non-zero when any `error` severity finding is present. A later strict mode can add stricter exit behavior in a separate design.

## Error Handling

- Use stable error/diagnostic codes for new commands and doctor output.
- Keep existing user-visible error strings stable unless tests are intentionally updated.
- Import must fail before writing when input is invalid unless `--merge` can skip rejected records safely and report them.
- `--dry-run` operations must not mutate store/cache/output paths.
- Unknown config keys should be surfaced by doctor and rejected by registry-backed config mutation.

## Testing Strategy

### Unit/Component Tests

- Registry lookup, uniqueness, aliases, and duplicate-key prevention.
- Config key get/set/remove via registry callbacks.
- Target/template/capability registry consistency.
- Subscription import/export parse and merge rules.
- Diagnostic probe output codes and severities.
- Service-layer filtering, batch edit, prune, and check behavior.

### Smoke Tests

- `sub import` then `sub list --json` shows imported subscriptions.
- `sub export --file` then `sub import --file` round-trips supported fields.
- `sub check --json` reports fetch/parse status.
- `sub prune --dry-run` reports without mutation.
- `doctor --json` includes workspace/config/template/asset/core/subscription/export readiness sections.
- help/completion includes registry-backed commands, config keys, targets, and options.

### Build Verification

Use project standard checks:

```bash
cmake -S . -B build && cmake --build build -j
ctest --test-dir build --output-on-failure
```

Also verify package generation:

```bash
cmake --build build --target package
```

## Acceptance Criteria

1. New subscription lifecycle commands are documented and covered by tests.
2. Config keys are registry-backed for list/get/set/remove, help, and completion.
3. Export targets and template slots are registry-backed enough that adding/removing a target has one clear edit path plus tests.
4. Capability behavior is centralized and tested for strict and non-strict export flows.
5. `doctor --json` reports structured readiness for workspace, config, profile, templates, assets, cores, subscriptions, and export targets.
6. `src/main.cpp` no longer owns long config/export/subscription implementation branches for touched commands.
7. Release validation workflow exists and runs build, tests, and package generation.
8. Standard build/test/package commands pass.

## Risks And Mitigations

- Risk: Big-bang refactor breaks CLI behavior.
  - Mitigation: lock existing CLI contracts first, then migrate one command family at a time.
- Risk: Registry abstractions become over-engineered.
  - Mitigation: use plain structs/functions, no dynamic plugin ABI in v0.2.5.
- Risk: Docs drift from registries.
  - Mitigation: add tests that compare registry keys/options against docs/help/completion.
- Risk: Subscription import corrupts existing store.
  - Mitigation: validate full import plan before write and support `--dry-run` or atomic write behavior.
- Risk: Large release takes too long.
  - Mitigation: implement in slices: registry foundation, config migration, target/capability migration, subscription lifecycle, diagnostics, release validation.

## Implementation Boundaries

- Prefer minimal, explicit C++ structs over generic framework code.
- Keep persistent data compatibility unless a migration is explicitly designed and tested.
- Keep CLI names and existing output contracts stable unless the implementation plan calls out a test-backed change.
- Do not introduce hidden global mutable registries; registries should be deterministic and easy to inspect in tests.
- Do not add a new protocol or target unless the full extension checklist is implemented.
