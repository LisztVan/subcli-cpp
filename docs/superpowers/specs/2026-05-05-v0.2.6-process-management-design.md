# v0.2.6 Process Management Design

## Goal

v0.2.6 completes the optional local process-management workflow around `run`, `restart`, `status`, `stop`, `daemon`, and logs. The release keeps configuration export as the primary product workflow, but makes local core hosting practical when users choose it.

The core outcome is a closed loop: start a core in the background, inspect its status, restart or stop it, and read its logs through `subcli` without manually discovering state files.

## Scope

- Keep `subcli run <target>` as a background managed process by default.
- Add `subcli run <target> --foreground` for direct interactive execution.
- Add `--log-file PATH` to `run` and `restart` for managed background runtime logs.
- Persist runtime metadata needed by status/log commands: pid, target, binary path, config path, log path, and start time.
- Add a root `subcli logs <mihomo|sing-box|xray|daemon>` command with `--tail N`, `--follow`, and `--file PATH`.
- Show log path and start time in `subcli status` output.
- Reuse existing daemon log handling for `subcli logs daemon`.
- Update completion, help, README, and tests.

## Non-Goals

- No cross-platform process supervisor guarantee. Runtime and daemon commands remain optional helpers.
- No terminal UI or web UI.
- No log rotation.
- No reliable last-exit-code tracking for background cores in this release. A forked detached process cannot be reaped later without a supervisor process; adding one is a larger design.
- No tee behavior for `--foreground --log-file`. Foreground mode writes to the terminal like the native core.

## CLI Design

### Runtime Commands

`subcli run <mihomo|sing-box|xray> [--file PATH] [--output-dir DIR] [--foreground] [--log-file PATH]`

`subcli restart <mihomo|sing-box|xray> [--file PATH] [--output-dir DIR] [--foreground] [--log-file PATH]`

Default `run` and `restart` start a managed background process. stdout and stderr append to a log file. Default log path is `state/runtime/<normalized-target>.log`, where `sing-box` becomes `sing_box.log` for consistency with existing runtime state filenames. `--log-file` overrides this path.

`--foreground` runs the core attached to the terminal and waits for it to exit. It does not write runtime state because there is no managed process to inspect after the command returns. `restart --foreground` stops any managed background process first, then runs the core in the foreground.

### Status Command

`subcli status [mihomo|sing-box|xray]`

For a running process, status prints target, `running`, pid, config path, log path, and start time. For stopped targets, status prints `stopped`; if stale state exists, status removes stale state as today, but logs remain on disk.

### Logs Command

`subcli logs <mihomo|sing-box|xray|daemon> [--tail N] [--follow] [--file PATH]`

`logs` chooses the path in this order:

1. `--file PATH` if provided.
2. Runtime state `log_path` for core targets.
3. Default runtime log path for core targets.
4. Daemon state custom log path for `daemon`, else default daemon log path.

`--tail N` prints the last N lines. Default is `200`. `--tail 0` prints the whole file. `--follow` prints current tail, then streams appended data until interrupted. If the log file does not exist, the command fails with a direct message telling the user to run the target first or pass `--file`.

## Architecture

### Runtime Metadata

Extend `RuntimeStatus` with:

- `logPath`
- `startedAt`

Update `loadRuntimeState` and `saveRuntimeState` to read and write these fields. Missing fields from old state files default to empty strings.

Add helpers in `runtime_process`:

- `runtimeLogPathForTarget(stateDir, target)`
- path normalization shared with `runtimeStatePathForTarget`

### Runtime Launch

Extend `startCoreRuntime` with `logPath`. Background child redirects stdin to `/dev/null` and stdout/stderr to the configured log file. Parent creates parent directories, writes state after fork, and reports pid/config/log.

Add a foreground launch path in CLI-level runtime code. It uses the same binary and core args resolution as background mode, but starts the core attached to the terminal and waits for completion. This stays in `main.cpp` unless it grows beyond the current command shape.

### Logs Reading

Implement small file utilities for log output:

- `printLogTail(path, lineCount)`
- `followLog(path)`

Keep this minimal and dependency-free. `--tail` can read the file line-by-line into a bounded deque for simplicity; log files are expected to be modest in this release. `--follow` polls file size and reads new data at short intervals.

### Daemon Logs

`logs daemon` uses `inspectDaemonProcess` to discover persisted custom log path. If state is absent or has no custom log path, it falls back to the existing default daemon log path.

## Error Handling

- Invalid targets return usage errors.
- Missing config file or missing core binary keeps current error behavior.
- Background launch fails if log parent directory cannot be created or log file cannot be opened.
- Duplicate background `run` remains rejected unless the user uses `restart`.
- Missing log files return an error and do not wait for future file creation.
- `--tail` must be a non-negative integer.
