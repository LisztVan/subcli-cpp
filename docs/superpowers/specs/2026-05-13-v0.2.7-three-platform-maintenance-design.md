# v0.2.7 Three-Platform Maintenance Design

## Scope

v0.2.7 is a cross-platform systems-maintenance release for Linux, macOS, and Windows. It may substantially reorganize internals because the project is still pre-1.0. The release goal is that users on all three platforms can build, test, package, and use the primary CLI without obvious platform-specific friction.

The release includes full platform-specific process-management support rather than only returning unsupported errors on Windows. Process execution, background runtime management, daemon management, PID inspection, executable discovery, current executable path discovery, and local test networking become explicit platform services with POSIX and Win32 implementations.

## Non-Goals

- No new proxy protocol or export target is added.
- No GUI, service installer, launchd unit, or Windows service is added.
- Existing profile/export semantics remain compatible unless tests identify platform path bugs that require normalization.
- External cores are still not bundled; users configure core paths or provide cores on `PATH`.

## Platform Support Contract

The following commands must be usable on Linux, macOS, and Windows after installation:

- `subcli --help`
- `subcli init`
- `subcli doctor --json`
- `subcli config list|get|set|remove`
- `subcli sub list|add|edit|remove|enable|disable|update|validate|import|export|check|prune`
- `subcli profile list|get|validate|explain`
- `subcli template list|get|set|reset|validate`
- `subcli asset list|status|validate|update`
- `subcli export all|mihomo|sing-box|xray`
- `subcli check mihomo|sing-box|xray`
- `subcli run|restart|stop|status|logs`
- `subcli daemon once|run|start|stop|status`
- `subcli completion bash`

Process-management commands are implemented with POSIX primitives on Linux/macOS and Win32 primitives on Windows. Behavior may differ in signal mechanics, but user-visible state, log files, process IDs, duplicate-start rejection, stale-state cleanup, and stop/status behavior must have parity.

## Architecture

Add a small platform boundary under `include/subcli/platform.hpp` and `src/platform_*.cpp`. The rest of the codebase calls portable functions for process execution, process start/stop, PID running checks, executable-file checks, current executable path detection, and local TCP test sockets.

Use compile-time file selection in CMake:

- `src/platform_posix.cpp` for Linux/macOS.
- `src/platform_windows.cpp` for Windows.
- `src/platform_common.cpp` for shared quoting/path helpers if needed.

The only direct includes of `<unistd.h>`, `<sys/wait.h>`, `<signal.h>`, `<fcntl.h>`, `<windows.h>`, and WinSock headers should be in platform implementation files and the test-only socket helper. Existing business modules such as `core_check.cpp`, `core_runtime.cpp`, `daemon.cpp`, `daemon_process.cpp`, `runtime_process.cpp`, and `main.cpp` should not directly own OS primitives.

## Process Model

### Foreground Process

`runProcessForeground(binary, args)` runs a child with inherited stdin/stdout/stderr and returns the child exit code. POSIX uses `fork`/`execv`/`waitpid`. Windows uses `CreateProcessW` and `WaitForSingleObject`.

### Captured Process With Timeout

`runProcessCapture(binary, args, timeoutSec)` captures stdout/stderr into one string and terminates the child on timeout. POSIX uses pipes and non-blocking reads. Windows uses inherited pipe handles and a reader thread.

### Background Process With Log Redirection

`startBackgroundProcess(binary, args, logPath)` starts a detached process, redirects stdout/stderr to the log file, and returns a portable process id. POSIX uses `fork`, `setsid`, `dup2`, and `/dev/null` stdin. Windows uses `CreateProcessW` with `DETACHED_PROCESS | CREATE_NEW_PROCESS_GROUP` and inheritable handles.

### Stop And Status

`isProcessRunning(pid)` reports whether a process exists and has not exited. POSIX uses `kill(pid, 0)`. Windows uses `OpenProcess` and `GetExitCodeProcess`.

`terminateProcess(pid, timeoutSec)` requests termination and waits. POSIX sends `SIGTERM`, waits, then sends `SIGKILL`. Windows uses `TerminateProcess` and waits for the process handle to signal. This is intentionally forceful because the managed core/daemon contract is lifecycle control, not graceful app-level shutdown.

## Testing Strategy

CTest must run on all three platforms. The C++ test binary is enabled on Windows and adapted to avoid POSIX-only includes and hardcoded `/bin` paths. POSIX-only bash smoke coverage remains Linux/macOS-only. A new CMake-script CLI smoke test runs on all platforms.

Required verification gates:

- Linux local: configure, build, CTest, package, package verification.
- macOS CI: configure, build, CTest, package.
- Windows CI: configure, build, CTest, package.
- Release workflow runs tests on all three platforms before packaging artifacts.
- Release-validation workflow uses a three-platform matrix.

## Documentation

Update `README.md`, `README.subcli.md`, and `docs/config-file.md` if path behavior changes. Document the v0.2.7 three-platform process-management contract and the fact that daemon/runtime management is implemented with platform-native process primitives rather than service managers.

## Acceptance Criteria

- `project(subcli VERSION 0.2.7 ...)` is set.
- `ctest --test-dir build --output-on-failure` passes locally on Linux.
- GitHub Actions release and release-validation workflows test Linux, macOS, and Windows.
- Windows builds and runs `subcli_tests` rather than skipping the test executable.
- `subcli run`, `status`, `logs`, `stop`, `restart`, and `daemon start|status|stop` are backed by platform APIs on Windows.
- `rg "unistd|fork|execv|waitpid|kill\(|CreateProcess|GetModuleFileName|winsock|socket\(" src include tests` shows OS primitives confined to platform implementation files and test socket helpers, except documentation/history files.
